# UIgod Decisions

This document records current product decisions.

## Confirmed

### UIgod Is A Meta-Skill

UIgod does not primarily generate websites.

It generates or upgrades child skills that help other agents understand a reusable web style family.

### Child Skills Are Style References

Child skills are not fixed page-generation workflows.

They should provide strong style understanding, references, and code examples so other agents can apply the style in their own task context.

### DESIGN.md Is The Canonical Design Truth

Each child skill has one canonical design-system document: `DESIGN.md`.

That file holds the durable style truth and should stay compatible with `awesome-design-md` style entries.

### SKILL.md Is The Operational Wrapper

`SKILL.md` should stay short.

It tells an agent when to use the child skill, what files to read first, and what parts of the style must be preserved.

It should not duplicate the whole design core.

### Web First

The first version only targets web pages and web interfaces.

Mobile, native apps, decks, and other surfaces are out of scope for v1.

### Single Style Family

Each child skill should represent one coherent style family.

Do not merge unrelated style directions into one child skill.

### Inputs Are Flexible

The user can provide:

- URLs
- screenshots
- copied code
- text descriptions

The number of references is decided by the user.

### Screenshot Support Is Optional

Screenshots can improve style understanding when available.

UIgod must not depend on screenshot tooling because some agents cannot capture or inspect screenshots.

### Code Reference Is Required

Every child skill should include code references.

The references should be decomposed and representative, not just one large source dump.

### Synthesis Over Source Reconstruction

UIgod usually cannot access exact original source code.

It should synthesize representative code patterns that express the style instead of pretending to reproduce the source.

### Match Before Creating

Before creating a new child skill, UIgod should check whether an existing child skill is close enough to upgrade.

If close, propose upgrading.

If distinct, propose creating a new child skill.

Ask the user to confirm either direction.

### Unified Skeleton

Generated child skills should follow a shared skeleton:

- `DESIGN.md`
- `README.md`
- `SKILL.md`
- `preview.html`
- `references/source-observations.md`
- `references/source-screenshots-or-notes/`
- `references/code-patterns/`
- `examples/README.md`
- `examples/applied/`

### Richer Than awesome-design-md

UIgod should absorb the best part of `awesome-design-md`:

- one canonical `DESIGN.md`
- stable section structure
- consistent entry quality bar

But UIgod should keep more execution assets than a static design library:

- agent-facing `SKILL.md`
- visual preview surface
- code patterns
- source observation trail

### Preview Is A Token Specimen Page

`preview.html` renders the `DESIGN.md` tokens and core motifs — color swatches, type specimens, shape samples, and style-defining visual devices — using the style's own fonts, colors, and radii.

Reason: an earlier iteration made the preview a separate low-fidelity page re-drawing, which drifted badly from the real examples. A tabbed iframe viewer fixed the drift but duplicated what opening `examples/` already gives. The token specimen page is the only content with no other home, so the preview keeps exactly that and nothing else.

### Source Evidence Lives In References

Source screenshots, copied public notes, page text, URL/date records, and viewport notes belong under `references/source-screenshots-or-notes/`.

This folder is evidence only.

Do not use it as a page template.

### Applied Examples Are Post-Validation Artifacts

`examples/applied/` contains examples generated by using the child skill on fresh prompts, after visual confirmation.

Applied examples should demonstrate transfer: same style family, new information architecture.

Do not add source-site clones or reconstructions as examples.

If the user does not specify a validation target, the default applied example is a fictional personal portfolio at `examples/applied/portfolio/index.html`.

Reason: a portfolio naturally exercises hero, navigation, case studies, capabilities, work history, CTA, responsive behavior, and style-defining motifs without encouraging source-site cloning.

### Testing Has Two Layers

UIgod tests should separate two questions:

1. did the generated child skill capture the reference style?
2. can the generated child skill apply that style to future requests?

`DESIGN.md` plus `preview.html` answer the first question by showing whether the style tokens, motifs, and rules were captured.

An applied example answers the second question by asking the child skill to generate a different topic in the same style.

## Deferred

The following are useful but not current priorities:

- formal style index
- automated similarity scoring
- heavy extraction scripts
- mandatory screenshot capture
- non-web child skills

## Current Development Direction

Build a working UIgod root skill and one reusable child-skill template first.

Then test the system by converting one real reference set into a child skill.
