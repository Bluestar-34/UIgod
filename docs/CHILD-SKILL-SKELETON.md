# Child Skill Skeleton

## 1. Purpose

Every child skill generated by UIgod should share one stable skeleton.

The skeleton is meant to preserve style understanding, not to enforce a rigid page-generation machine.

The canonical design source for every child skill is `DESIGN.md`.

## 2. Scope

For now, each child skill should:

- describe one style family
- target web work only
- help other agents understand the style quickly
- provide decomposed code references
- remain compatible with the `awesome-design-md` / Stitch-style `DESIGN.md` format

## 3. Baseline Structure

```text
child-skill-name/
├── DESIGN.md
├── README.md
├── SKILL.md
├── preview.html
├── references/
│   ├── source-observations.md
│   ├── source-screenshots-or-notes/
│   └── code-patterns/
└── examples/
    ├── README.md
    └── applied/
```

## 4. File Responsibilities

### `DESIGN.md`

This is the canonical design system document.

It should contain the core style truth in a structure compatible with `awesome-design-md`.

The default section model is:

- frontmatter with description and tokens
- `## Overview`
- `## Colors`
- `## Typography`
- `## Layout`
- `## Elevation & Depth`
- `## Shapes`
- `## Components`
- `## Do's and Don'ts`
- `## Responsive Behavior`
- `## Iteration Guide`
- `## Known Gaps`

If a style does not need much depth for one section, keep that section short rather than omitting the structure entirely.

### `README.md`

Contains a short human-facing explanation:

- what style family this is
- what it was derived from
- what files matter
- how to preview it quickly

### `SKILL.md`

Contains:

- name
- trigger description
- when to use it
- how to consult `DESIGN.md`
- how to consult examples and source notes
- what the agent should preserve during generation

This file should stay concise.

### `preview.html`

A token specimen page, not a page demo.

It renders the `DESIGN.md` tokens with the style's own fonts, colors, and radii:

- color swatches with token names and values
- type specimens for each typography token
- shape / radius samples
- optional links to confirmed applied examples when they exist

Rules:

- the preview must open locally over `file://` with no build step
- it must not duplicate or re-draw the source site's page layout
- it is the primary first-pass visual validation surface during skill creation
- specimen styling must use the same token values as the examples so the two never diverge
- it should follow the `getdesign.md` style of fast design-system scanning: tokens first, page structure second

### `references/source-observations.md`

Contains the reference audit trail:

- source URLs or screenshots
- **`## CSS Extraction` section** — documenting all extracted CSS tokens per `docs/REFERENCE-ANALYSIS-GUIDE.md`:
  - CSS custom properties
  - @font-face declarations
  - border-radius, border-width, box-shadow values mapped to usage
  - section background colors mapped
  - typography scale (all font-size declarations including clamp values)
  - interactive behaviors (hover, transitions, cursors)
  - patterns (navigation style, CTA style, etc.)
- observed patterns from the source
- what was preserved vs what was synthesized
- known uncertainty
- if screenshot-only: all tokens marked as `[inferred]`

### `references/source-screenshots-or-notes/`

Holds optional source evidence:

- user-provided screenshots
- agent-captured screenshots when available
- copied public text or style notes
- viewport, date, and URL notes

This folder is evidence only. Do not treat it as template code or a page structure source.

### `references/code-patterns/`

Holds synthesized, reusable component-scale implementation references.

Good code patterns:

- `nav.html`
- `hero.html`
- `section.html`
- `list-row.html`

These should be representative, not exhaustive. They are references, not validation examples and not page templates.

### `examples/applied/`

Holds confirmed examples generated by using this child skill on fresh web prompts.

Applied examples are not required during the initial style extraction phase.

Add them after the skill has produced a result that a human or visual QA has accepted.

Good examples:

- `examples/applied/portfolio/`
- `examples/applied/product-launch/`
- `examples/applied/dashboard/`

Applied examples must use the extracted style with a new information architecture appropriate to the requested site type.

Do not place source-site replicas here.

Default validation example:

- If the user does not specify a validation target, generate `examples/applied/portfolio/index.html`.
- Use a fictional but realistic personal portfolio.
- Include hero, navigation, project cases, capability/workflow section, experience or notes, contact CTA, responsive behavior, and at least one style-defining motif.
- The page must prove transfer, not resemblance to the source site's layout.

This does not replace decomposed code patterns.

## 5. Authoring Rules

- Keep one child skill tied to one style family.
- Treat `DESIGN.md` as the single design truth source.
- Prefer web-native examples over abstract moodboard language.
- Prefer synthesized code over fake source reconstruction.
- Do not clone the source site as an example.
- If the original framework is recognizable, it may inform examples, but style remains primary.
- If the user only provides screenshots, the child skill must still be able to express the style.
- Do not require screenshots to exist for the child skill to be useful.
- Keep `SKILL.md` instructional, not duplicative of `DESIGN.md`.

## 6. Minimum Quality Bar

A child skill is not ready unless:

- `DESIGN.md` can stand alone as a coherent style system document
- another agent can tell what the style is quickly
- another agent can restyle a fresh page from the package without needing the original site again
- the style does not read as generic premium UI language
- code patterns feel internally coherent
- the design guardrails are sharp enough to prevent obvious drift
- `preview.html` lets a human scan the palette, type system, and shapes at a glance
- any applied examples use fresh structure and content while preserving the style family

## 7. Future Extensions

Possible later additions:

- `assets/user-supplied/`
- `preview-dark.html`
- `preview.png`
- `catalog.json`
- lightweight helper tools for repeated formatting or packaging

These are optional and should be added only when real usage justifies them.
